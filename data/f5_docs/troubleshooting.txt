# F5 BIG-IP Troubleshooting Guide

## Systematic Troubleshooting Approach

### Step 1: Define the Problem
- What is the expected behavior?
- What is the actual behavior?
- When did the problem start?
- What changed recently?

### Step 2: Gather Information
- Check logs
- Review configuration
- Verify network connectivity
- Test from different locations

### Step 3: Isolate the Issue
- Determine if issue is BIG-IP, server, or network related
- Test each component individually
- Use bypass methods to confirm

### Step 4: Implement Solution
- Make one change at a time
- Document all changes
- Verify the fix

## Common Issues and Solutions

### Virtual Server Issues

#### Virtual Server Showing "Unknown" Status
Causes:
- No monitor assigned
- All pool members down
- Pool not assigned

Solutions:
```bash
# Check pool assignment
tmsh list ltm virtual my_vs pool

# Check pool members
tmsh show ltm pool my_pool members

# Assign monitor to pool
tmsh modify ltm pool my_pool monitor http
```

#### Traffic Not Reaching Virtual Server
Causes:
- Routing issue
- Firewall blocking
- Virtual server disabled
- Wrong VLAN assignment

Solutions:
```bash
# Verify virtual server is enabled
tmsh show ltm virtual my_vs enabled

# Check VLAN assignment
tmsh list ltm virtual my_vs vlans

# Test connectivity
ping [virtual_server_ip]

# Verify ARP
tmsh show net arp
```

### Pool and Member Issues

#### Pool Members Marked Down

Pool members and nodes are marked offline or down when:
- No response arrives within the specified timeout period
- Health monitor fails to receive expected response
- Performance degrades below thresholds
- Response doesn't match expected content criteria (for ECV monitors)
- Network connectivity is lost between BIG-IP and pool member

**Step 1: Check Pool Member Status**
```bash
# View pool member status
tmsh show ltm pool my_pool members

# View detailed status with field formatting
tmsh show ltm pool all members field-fmt | grep -P "(ltm\ pool|active-member-cnt|addr|monitor-status)"

# Check how many members are in "checking" status
tmsh show ltm pool all members field-fmt | grep "monitor-status\ checking" | wc -l

# Search logs for pool member status changes
grep -i 'pool member' /var/log/ltm | grep 'status'
```

**Step 2: Verify Network Connectivity**
```bash
# Ping pool member from BIG-IP
ping [member_ip]

# Check ARP entry
arp -a | grep [member_ip]
tmsh show net arp | grep [member_ip]

# Verify VLAN and routing
tmsh list net vlan
tmsh list net route
```

**Step 3: Test Service Directly from BIG-IP**
```bash
# Test HTTP connectivity directly to pool member
curl -i http://[member_ip]:[port]/

# Test with verbose output
curl -v http://[member_ip]:[port]/health

# Test HTTPS
curl -k https://[member_ip]:[port]/
```

**Step 4: Check Monitor Configuration**

The interval/timeout ratio is critical. The timeout value should be equal to three times the interval value plus one. For example, the default ratio is 5/16 (3 × 5 + 1 = 16).

```bash
# View monitor configuration
tmsh list ltm monitor http my_monitor

# Check monitor send/receive strings (ECV monitors)
tmsh list ltm monitor http my_monitor send recv

# View all monitors assigned to pool
tmsh list ltm pool my_pool monitor
```

**Step 5: Packet Capture for Monitor Traffic**
```bash
# Capture health monitor traffic to pool member
tcpdump -ni [server_vlan] host [member_ip] and port [member_port] -w /var/tmp/monitor_capture.pcap

# Capture with verbose output
tcpdump -ni [server_vlan] -Xvvs0 host [member_ip]

# View the capture
tcpdump -r /var/tmp/monitor_capture.pcap
```

**Step 6: Check the bigd Process**

The bigd process manages health checking for pool members, nodes, and services. If bigd fails, health check status remains in current state until bigd restarts.

```bash
# Check if bigd is running
ps aux | grep bigd

# Restart bigd if needed (use with caution)
bigstart restart bigd

# Check bigd logs
grep bigd /var/log/ltm
```

**Step 7: Verify Send/Receive Strings (ECV Monitors)**

For Extended Content Verification (ECV) monitors like HTTP/HTTPS, the Send string must be properly constructed or the server may send an unexpected response.

Common Send string issues:
- Missing HTTP version: Use `GET /path HTTP/1.1\r\nHost: hostname\r\n\r\n`
- Wrong path: Verify the health check URL exists on the server
- Missing headers: Some servers require Host header

```bash
# Test the exact send string manually
echo -e "GET /health HTTP/1.1\r\nHost: example.com\r\n\r\n" | nc [member_ip] [port]
```

**Step 8: Check Manual Resume Setting**

If Manual Resume is set to Yes, the pool member remains unavailable until manually re-enabled, even after the server recovers.

```bash
# Check manual resume setting
tmsh list ltm monitor http my_monitor manual-resume

# Manually enable a pool member
tmsh modify ltm pool my_pool members modify { [member_ip]:[port] { state user-up } }
```

**Common Monitor Issues and Solutions:**

| Issue | Cause | Solution |
|-------|-------|----------|
| Members flapping up/down | Timeout too short | Increase timeout (3x interval + 1) |
| All members down | Wrong receive string | Verify expected response matches |
| Members stuck in checking | bigd process issue | Restart bigd |
| Intermittent failures | Network latency | Increase interval and timeout |
| HTTPS monitor failures | Certificate issues | Use ssl-profile or disable cert verification |

#### Uneven Load Distribution
Causes:
- Persistence settings
- Slow ramp time
- Priority groups
- Member weights

Solutions:
```bash
# Check persistence
tmsh list ltm virtual my_vs persist

# Review pool settings
tmsh list ltm pool my_pool load-balancing-mode slow-ramp-time

# Check member statistics
tmsh show ltm pool my_pool members field-fmt
```

### SSL/TLS Issues

#### SSL Handshake Failures
Causes:
- Certificate issues
- Cipher mismatch
- Protocol version mismatch
- Certificate chain incomplete

Solutions:
```bash
# Test SSL connection
openssl s_client -connect [ip]:443

# Check certificate
tmsh list sys crypto cert [cert_name]

# Verify cipher configuration
tmsh list ltm profile client-ssl my_ssl ciphers

# Check for SSL errors
grep -i ssl /var/log/ltm
```

#### Certificate Chain Issues
```bash
# View certificate chain
openssl s_client -connect [ip]:443 -showcerts

# Verify chain order
openssl verify -CAfile chain.crt server.crt
```

### Performance Issues

#### High CPU Usage
Causes:
- Traffic volume
- iRule complexity
- SSL processing
- Logging overhead

Solutions:
```bash
# Check CPU by process
top

# Check TMM CPU specifically
tmsh show sys tmm-info

# Review iRule performance
tmsh show ltm rule [rule_name] stats

# Check SSL performance
tmsh show sys performance ssl
```

#### Memory Issues
```bash
# Check memory usage
tmsh show sys memory

# Check TMM memory
tmsh show sys tmm-info

# Check connection table size
tmsh show sys connection count

# Clear old connections if needed
tmsh delete sys connection all
```

### Connection Issues

#### Connections Timing Out
Causes:
- SNAT not configured
- Routing asymmetry
- Timeout values too low
- Server not responding

Solutions:
```bash
# Check SNAT configuration
tmsh list ltm virtual my_vs source-address-translation

# Verify routing
traceroute [server_ip]

# Check timeout settings
tmsh list ltm profile tcp my_tcp idle-timeout

# Monitor connections
tcpdump -ni [vlan] host [client_ip]
```

#### Connections Refused
```bash
# Check virtual server status
tmsh show ltm virtual my_vs availability

# Check connection limits
tmsh list ltm pool my_pool members { connection-limit }

# Verify port availability
netstat -an | grep [port]
```

### iRule Troubleshooting

#### Enable iRule Logging
```tcl
when HTTP_REQUEST {
    log local0. "Client: [IP::client_addr] URI: [HTTP::uri]"
}
```

#### Check iRule Timing
```bash
# Enable timing statistics
tmsh modify ltm profile http my_http insert-xforwarded-for enabled

# View iRule statistics
tmsh show ltm rule my_rule stats
```

#### Common iRule Errors
- TCL syntax errors - Check for missing brackets, quotes
- Event misuse - Verify correct event for commands
- Variable scope - Use `set` appropriately

### High Availability Issues

#### Sync Failures
```bash
# Check sync status
tmsh show cm sync-status

# View sync errors
cat /var/log/ltm | grep -i sync

# Force sync
tmsh run cm config-sync to-group [group_name]
```

#### Failover Problems
```bash
# Check failover status
tmsh show cm device [device_name]

# View failover objects
tmsh list cm traffic-group

# Check gateway failsafe
tmsh list ltm gateway-icmp
```

## Packet Capture

### Basic Capture
```bash
# Capture traffic on specific VLAN
tcpdump -ni [vlan_name] host [ip_address] -w /var/tmp/capture.pcap

# Capture specific port
tcpdump -ni [vlan_name] port 443

# Capture with verbose output
tcpdump -ni [vlan_name] -Xvvs0 host [ip_address]
```

### Internal Capture (includes modified traffic)
```bash
# Capture on internal interface
tcpdump -ni 0.0:nnnp host [ip_address]
```

### Analysis
```bash
# Read capture file
tcpdump -r /var/tmp/capture.pcap

# Download for Wireshark analysis
scp admin@[bigip_ip]:/var/tmp/capture.pcap ./
```

## Best Practices

1. **Regular Health Checks**
   - Monitor system performance
   - Review logs daily
   - Check for software updates

2. **Documentation**
   - Keep configuration records
   - Document changes
   - Maintain runbooks

3. **Backup Strategy**
   - Regular UCS backups
   - Test restoration procedures
   - Store backups securely

4. **Change Management**
   - Test in non-production first
   - Schedule maintenance windows
   - Have rollback plans

5. **Monitoring**
   - Implement SNMP monitoring
   - Set up alerting thresholds
   - Track key performance metrics
