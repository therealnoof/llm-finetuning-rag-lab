# F5 BIG-IP SSL/TLS Traffic Management Guide

## Introduction

SSL/TLS offloading transfers the cryptographic processing of encrypted traffic from backend servers to BIG-IP. The BIG-IP system processes SSL handshakes normally performed by destination servers, decrypting client requests and encrypting responses. This offloads computational burden from backend servers while enabling traffic inspection and manipulation.

In BIG-IP 17.0, TLS 1.3 is supported with RFC 8446. The system handles protocol negotiation based on client capabilities and configured cipher suites.

## SSL Offloading Architectures

### Client SSL Termination (SSL Offloading)
BIG-IP terminates SSL from clients and sends unencrypted traffic to servers.

Traffic flow:
1. Client connects via HTTPS (encrypted)
2. BIG-IP decrypts traffic using Client SSL profile
3. BIG-IP sends HTTP (unencrypted) to servers
4. Server responds via HTTP
5. BIG-IP encrypts response
6. Encrypted response sent to client

Benefits:
- Reduced server CPU load
- Centralized certificate management
- Traffic inspection capability
- iRule processing possible

### SSL Bridging (Re-encryption)
BIG-IP terminates client SSL and creates new SSL connections to servers.

Traffic flow:
1. Client connects via HTTPS
2. BIG-IP decrypts (Client SSL profile)
3. BIG-IP re-encrypts (Server SSL profile)
4. Server receives encrypted traffic
5. Response follows reverse path

Use cases:
- End-to-end encryption required
- Different certificates for internal/external
- Different cipher requirements

### SSL Passthrough
BIG-IP forwards encrypted traffic without decryption.

Limitations:
- No traffic inspection
- No content manipulation
- Limited to L4 load balancing

## SSL Profile Architecture

### Client SSL Profiles
Enable the BIG-IP to decrypt inbound client traffic and encrypt outbound responses.

Key configurations:
- **Certificate Key Chains**: Support multiple key types (RSA, DSA, ECDSA) to negotiate appropriate cipher suites with different clients
- **Cipher Configuration**: Select predefined cipher groups or manually specify custom cipher strings
- **Client Authentication**: Options include Ignore (default), Request (optional), or Require (mandatory)
- **Session Caching**: Cache size defaults to 262,144 sessions; timeout defaults to 3,600 seconds

Creating a Custom Client SSL Profile:
1. Navigate to Local Traffic > Profiles > SSL > Client
2. Create profile with parent "clientssl" and enable Advanced configuration
3. Define Certificate Key Chains with minimum RSA certificate/key pair
4. Configure cipher settings (group or custom string)
5. Assign to virtual server

TMSH Configuration:
```bash
# Create Client SSL profile with certificate key chain
tmsh create ltm profile client-ssl my_clientssl {
    cert-key-chain {
        my_chain {
            cert my_cert.crt
            key my_cert.key
            chain my_intermediate.crt
        }
    }
    ciphers 'DEFAULT'
    options { dont-insert-empty-fragments no-tlsv1 no-tlsv1.1 }
}
```

### Server SSL Profiles
Enable encryption/decryption for server-side connections to backend servers.

Key configurations:
- **Server Authentication**: Verify backend certificate validity before establishing connections
- **Certificate Validation**: Check expiration, trust status, and revocation via CRL or OCSP
- **Proxy SSL**: Allow direct client-server certificate authentication through the BIG-IP intermediary
- **Server Name Indication (SNI)**: Specify hostname for servers hosting multiple certificates

Creating a Custom Server SSL Profile:
1. Access Local Traffic > Profiles > SSL > Server
2. Select parent "serverssl" with Advanced configuration
3. Specify certificate, key, and chain from system certificates
4. Configure server authentication requirements
5. Define certificate validation parameters

TMSH Configuration:
```bash
# Create Server SSL profile with server authentication
tmsh create ltm profile server-ssl my_serverssl {
    ca-file trusted_ca_bundle.crt
    server-name www.example.com
    authenticate require
    authenticate-depth 3
}
```

## Certificate Management

### Certificate Key Chains
BIG-IP supports multiple key types in a single Client SSL profile:
- RSA certificates (2048-bit or higher recommended)
- DSA certificates
- ECDSA certificates (faster performance)

This allows negotiation of appropriate cipher suites based on client capabilities.

### Certificate and Key Import
```bash
# Import certificate
tmsh install sys crypto cert my_cert from-local-file /var/tmp/cert.crt

# Import private key
tmsh install sys crypto key my_key from-local-file /var/tmp/key.key

# Import certificate bundle/chain
tmsh install sys crypto cert my_chain from-local-file /var/tmp/chain.crt

# Verify certificate details
tmsh list sys crypto cert my_cert
```

### Certificate Chain Configuration
Proper chain order (bottom to top):
1. Server certificate
2. Intermediate CA(s)
3. Root CA (optional - usually in client trust store)

Use certificate chains containing intermediate CA certificates for complete validation paths.

### Certificate Best Practices for High Availability
- Always use non-default certificate names in Device Service Clustering (DSC) Sync-Failover configurations
- Ensure identical certificate specifications across device group instances to enable successful failover
- Protect passphrases; the system automatically encrypts them

## Cipher Configuration

### Cipher Groups vs Custom Strings
BIG-IP offers two approaches:
1. **Cipher Groups**: Predefined, managed sets of ciphers (recommended)
2. **Custom Cipher Strings**: Manual specification for specific requirements

### Cipher String Syntax
```
# Format: keyword[:keyword]
TLSv1.2          # TLS 1.2 ciphers only
TLSv1.3          # TLS 1.3 ciphers only
!SSLv3           # Exclude SSL 3.0
!ADH             # Exclude anonymous DH
!EXPORT          # Exclude export ciphers
!NULL            # Exclude null ciphers
@STRENGTH        # Order by strength
```

### Recommended Cipher Strings

Modern (TLS 1.2+ with TLS 1.3):
```
TLSv1.2+TLSv1.3:!aNULL:!eNULL:!EXPORT:!DES:!RC4:!3DES:!MD5:!PSK
```

Compatibility (wider client support):
```
ECDHE+AESGCM:DHE+AESGCM:ECDHE+AES:DHE+AES:!aNULL:!MD5:!DSS
```

### View Available Ciphers
```bash
tmsh list ltm profile client-ssl my_clientssl ciphers
tmsh run util bash -c "openssl ciphers -v 'CIPHER_STRING'"
```

## Client Certificate Authentication

Mutual TLS (mTLS) requires clients to present certificates for authentication.

### Authentication Modes
- **Ignore** (default): Don't request client certificate
- **Request**: Request certificate but don't require it
- **Require**: Require valid client certificate to proceed

### Configuration
```bash
# Require client certificate
tmsh modify ltm profile client-ssl my_clientssl {
    client-cert-ca ca_bundle.crt
    peer-cert-mode require
}
```

### Access Client Certificate in iRules
```tcl
when CLIENTSSL_CLIENTCERT {
    set subject [SSL::cert subject]
    set issuer [SSL::cert issuer]
    log local0. "Client cert subject: $subject"
}
```

### Client Certificate Constrained Delegation (C3D)
When enabled (default), simplifies credential requirements through certificate-based delegation. The Generic Alert setting causes the system to send all SSL alerts using a generic handshake failure message for security.

## Advanced Features

### OCSP Stapling
Certificates include revocation status without client-side lookups, improving performance by eliminating the need for clients to contact OCSP responders.

```bash
# Create OCSP stapling profile
tmsh create ltm profile ocsp-stapling my_ocsp {
    dns-resolver my_resolver
    sign-hash sha256
}

# Enable in client SSL profile
tmsh modify ltm profile client-ssl my_clientssl {
    ocsp-stapling enabled
    status-age 86400
}
```

### Proxy SSL
Allows direct client-server certificate authentication through the BIG-IP intermediary, enabling:
- Pass-through of client certificates to backend servers
- End-to-end certificate-based authentication
- Visibility into encrypted traffic while maintaining certificate chain

### Dynamic Record Sizing
Enable "Allow Dynamic Record Sizing" for improved TLS performance by adjusting record sizes based on connection state:
```bash
tmsh modify ltm profile client-ssl my_clientssl {
    allow-dynamic-record-sizing enabled
}
```

### Session Management
Configure based on workload patterns:
```bash
tmsh modify ltm profile client-ssl my_clientssl {
    cache-size 262144
    cache-timeout 3600
    session-ticket enabled
}
```

### Renegotiation Settings
Balance security with throughput by configuring:
- Renegotiation period
- Renegotiation size
- Maximum renegotiation records

## TLS Version Configuration

### Recommended Settings
```bash
# Client SSL profile - disable older versions, enable TLS 1.3
tmsh modify ltm profile client-ssl my_clientssl {
    options { dont-insert-empty-fragments no-tlsv1 no-tlsv1.1 }
}
```

Version settings:
- no-sslv2 - Disable SSL 2.0 (always disable)
- no-sslv3 - Disable SSL 3.0 (recommended)
- no-tlsv1 - Disable TLS 1.0 (recommended)
- no-tlsv1.1 - Disable TLS 1.1 (recommended)
- TLS 1.2 and TLS 1.3 - Enable (recommended)

## Perfect Forward Secrecy (PFS)

Ensures session keys can't be compromised if server key is leaked.

Enable by using ECDHE or DHE cipher suites:
```bash
tmsh modify ltm profile client-ssl my_clientssl {
    ciphers 'ECDHE+AESGCM:ECDHE+AES:DHE+AESGCM:DHE+AES'
}
```

## Performance Optimization

### Hardware Acceleration
BIG-IP platforms include SSL acceleration hardware. Monitor performance:
```bash
tmsh show sys performance ssl
```

### Optimization Tips
1. Enable session caching or session tickets
2. Use ECDSA certificates (faster than RSA)
3. Prefer ECDHE over DHE for key exchange
4. Use appropriate key sizes (RSA 2048 sufficient for most cases)
5. Enable OCSP stapling
6. Consider TLS 1.3 for improved handshake performance
7. Enable dynamic record sizing
8. Select appropriate hash algorithms (SHA256, SHA384)

## Troubleshooting

### Common Issues
1. Certificate chain incomplete - Add intermediate certificates
2. Certificate mismatch - Verify hostname matches certificate CN/SAN
3. Cipher mismatch - Check client/server cipher overlap
4. Protocol mismatch - Enable compatible TLS versions

### Debug Commands
```bash
# Check SSL handshake details
tmsh run util tcpdump -ni external -Xvv port 443

# View SSL statistics
tmsh show ltm profile client-ssl my_clientssl

# Check certificate expiration
openssl s_client -connect host:443 2>/dev/null | openssl x509 -noout -dates

# Test SSL configuration
openssl s_client -connect host:443 -tls1_2
openssl s_client -connect host:443 -tls1_3
```

## Best Practices

1. Use TLS 1.2 or higher only (TLS 1.3 recommended)
2. Implement strong cipher suites with cipher groups
3. Enable Perfect Forward Secrecy (ECDHE preferred)
4. Maintain proper certificate chains with intermediates
5. Monitor certificate expiration proactively
6. Enable OCSP stapling for performance
7. Use appropriate key sizes (2048+ bit RSA, 256+ bit ECDSA)
8. Regularly update to address vulnerabilities
9. Use non-default certificate names in HA configurations
10. Test SSL configuration with external scanners
