# F5 BIG-IP Load Balancing Guide

## Introduction

Load balancing distributes incoming traffic across multiple servers to optimize resource utilization, maximize throughput, minimize response time, and ensure high availability. F5 BIG-IP provides sophisticated load balancing capabilities through its Local Traffic Manager (LTM) module.

## Load Balancing Methods

BIG-IP supports numerous load balancing algorithms to match different application requirements. These methods are categorized as either **Static** (do not use real-time traffic metrics) or **Dynamic** (use current server performance metrics to distribute traffic).

### Static Methods

Static load balancing methods do not use any traffic metrics from the node/pool member to distribute traffic. They follow predetermined algorithms.

#### Round Robin
Distributes requests in a circular and sequential pattern among pool members. Over time, each server receives an equal number of connections.
- Best for: Homogeneous server environments with similar capacity
- Pros: Simple, predictable, equal distribution over time
- Cons: Doesn't consider server load, capacity, or current performance
- Use when: All servers have equal performance capabilities

#### Ratio (Member)
Distributes requests proportionally based on assigned weight values. Higher ratios receive more traffic. Uses weighted round robin to distribute requests based on assigned priority levels.
- Best for: Servers with different capacities (e.g., newer vs older hardware)
- Example: A ratio of 3:2:1:1 means of 8 requests, first server receives 3, second receives 2, remaining servers each get 1
- Use when: You need to direct more traffic to more capable servers

#### Ratio (Node)
Similar to Ratio (Member) but weights are assigned at the node level, applying to all pool memberships for that node.

### Dynamic Methods

Dynamic load balancing methods use real-time traffic metrics from the node to distribute traffic based on current conditions.

#### Least Connections (Member)
Distributes traffic to the pool member currently handling the fewest active connections. Only active connections are counted; idle connections are not included in the calculation.
- Best for: Long-lived connections like FTP, SSH, or database connections
- Behavior: If all servers have the same number of connections, distribution falls back to round-robin
- Key difference from Round Robin: Adapts to actual server load in real-time
- Use when: Request processing times vary significantly or connections persist for extended periods

#### Least Connections (Node)
Considers connection count at the node level across all pools, useful when the same server participates in multiple pools.

#### Fastest (Node)
Routes connections to pool members based on server response time. Prioritizes servers that respond quickest.
- Best for: Applications where server performance varies
- Requires: Health monitor that measures response time (TTFB - Time to First Byte)
- Use when: Backend servers have varying performance characteristics

#### Fastest (Application)
Uses application-layer response times for routing decisions at Layer 7.

#### Observed (Member)
Combines connection count with response time for dynamic load distribution. Similar to Ratio but with automatically calculated ratios.
- Servers below average connection counts receive a ratio of 3
- Servers above average receive a ratio of 2
- Adapts to changing conditions automatically
- Use when: You want automatic ratio adjustment based on performance

#### Predictive (Member)
An aggressive version of Observed that analyzes performance trends to predict which server will perform best.
- Assigns ratios of 4 to underutilized servers
- Assigns ratios of 1 to overutilized servers
- Uses historical data for ranking
- Best for: Environments with fluctuating or predictable load patterns

#### Weighted Least Connections
Combines static weights with dynamic connection counting.
- Formula: (current_connections + 1) / weight
- Routes to member with lowest calculated value
- Use when: You need both capacity-based weighting and connection awareness

### Advanced Dynamic Methods

#### Round Trip Time (RTT)
Distributes requests to the pool member with the fastest measured round trip time between the data center and client. Optimizes for geographic performance and network latency.

#### Least Sessions
Optimizes for session-based workloads by tracking active sessions rather than connections.

#### Dynamic Ratio
Adjusts ratios dynamically based on probe data from servers. Treats performance values as proportions rather than absolute selections.

#### Packet Rate
Distributes requests to the pool member currently processing the fewest number of packets per second. Useful for identifying underutilized servers.

#### Kilobytes per Second
Routes requests to the pool member currently processing the fewest kilobytes per second. Requires BIG-IP to collect throughput metrics.

#### Completion Rate
Distributes requests to the pool member that maintains the least number of dropped or timed-out packets during transactions. Prioritizes reliability.

#### CPU
Selects pool members that currently have the most CPU processing time available. Prevents overloading processors.

#### Quality of Service (QoS)
Uses current performance metrics to calculate an overall QoS score for each pool member. Considers multiple factors simultaneously: RTT, hops, completion rate, packet rate, throughput, topology, capacity, and custom scores. Provides holistic load distribution.

### Choosing Between Round Robin and Least Connections

| Aspect | Round Robin | Least Connections |
|--------|-------------|-------------------|
| Type | Static | Dynamic |
| Metrics Used | None | Active connection count |
| Distribution | Equal/Sequential | Based on current load |
| Best For | Equal-capacity servers, short-lived requests | Long-lived connections (FTP, SSH, databases) |
| Overhead | Minimal | Requires connection tracking |
| Adaptability | None - ignores server state | Adapts to real-time conditions |

## Priority Groups

Priority groups enable tiered failover between server groups.

Configuration:
1. Assign priority values to pool members (higher = primary)
2. Set the pool's "Priority Group Activation" threshold
3. Lower priority members only receive traffic when higher priority group capacity falls below threshold

Example:
- Priority 2: Primary servers (data center A)
- Priority 1: Backup servers (data center B)
- Activation: Less than 1 member available

## Slow Ramp Time

Gradually increases traffic to newly available pool members.

Benefits:
- Allows server caches to warm up
- Prevents overwhelming just-recovered servers
- Smoother distribution of load

Configuration:
- Set ramp time in seconds
- New members start with reduced traffic
- Traffic increases linearly until ramp time expires

## Connection Limits

Protect servers from overload by limiting concurrent connections.

Settings:
- Pool member connection limit
- Node connection limit
- Ratio limit (connections per ratio unit)

When limit is reached:
- New connections queue or go to other members
- Optional: Configure action on pool down

## One Connect (Connection Optimization)

Reduces server connection overhead through connection reuse.

Benefits:
- Fewer TCP connections to servers
- Reduced server CPU and memory usage
- Improved response times

How it works:
1. BIG-IP maintains pool of connections to servers
2. Client requests use existing idle connections
3. Connections returned to pool after response

Configuration:
- Assign OneConnect profile to virtual server
- Set idle timeout and maximum connections
- Configure source mask for connection reuse scope

## Health Monitoring Integration

Load balancing relies on health monitors to determine member availability.

Considerations:
- Monitor interval affects responsiveness
- Use application-appropriate monitors
- Configure proper timeout values
- Consider monitor dependencies

Monitor impact on load balancing:
- Failed members excluded from rotation
- Recovered members added back (consider slow ramp)
- Monitor state affects availability calculations

## Persistence with Load Balancing

Persistence (session affinity) must balance with load distribution.

Interaction:
- Persistence overrides load balancing for existing sessions
- New sessions follow load balancing rules
- Session timeout affects rebalancing

Persistence methods:
- Source Address Affinity - Based on client IP
- Cookie Persistence - HTTP cookie-based
- SSL Session ID - TLS session-based
- Universal - Custom key using iRules

## GSLB (Global Server Load Balancing)

DNS-based load balancing across data centers using BIG-IP GTM/DNS.

Features:
- Geolocation routing
- Performance-based routing
- Disaster recovery
- Active-active data centers

GSLB methods:
- Round Robin
- Ratio
- Topology (location-based)
- Static Persist
- Global Availability

## Performance Considerations

1. **Method Selection**
   - Match algorithm to application behavior
   - Consider session length and request patterns
   - Start simple, optimize as needed

2. **Monitoring**
   - Track distribution across members
   - Monitor server utilization
   - Analyze response times

3. **Capacity Planning**
   - Size pools for peak load plus headroom
   - Plan for member failures
   - Consider burst traffic patterns

4. **Optimization**
   - Use appropriate health check intervals
   - Enable connection optimization where beneficial
   - Configure proper timeout values

## TMSH Commands

```bash
# Create pool with load balancing method
tmsh create ltm pool my_pool load-balancing-mode least-connections-member

# Add members with ratios
tmsh modify ltm pool my_pool members add { 10.0.0.1:80 { ratio 3 } 10.0.0.2:80 { ratio 1 } }

# Configure priority group
tmsh modify ltm pool my_pool members modify { 10.0.0.1:80 { priority-group 2 } }

# Set slow ramp time
tmsh modify ltm pool my_pool slow-ramp-time 30

# View pool statistics
tmsh show ltm pool my_pool members

# View load balancing decision statistics
tmsh show ltm pool my_pool field-fmt
```

## Best Practices

1. Choose load balancing methods based on actual application behavior
2. Test load distribution under realistic conditions
3. Monitor and adjust as application patterns change
4. Use priority groups for DR scenarios
5. Enable slow ramp for applications that benefit from warmup
6. Set appropriate connection limits to protect servers
7. Document load balancing configuration decisions
8. Regularly review pool statistics and adjust as needed
9. Consider seasonal or time-based traffic patterns
10. Test failover scenarios periodically
